1. 总流程：从事件到五层画像

重点：行为不会总是直接改画像，多数先进入偏好、候选或缓冲层。

2. 举例：七条流分别怎么改 soul

同一个动作，先发生什么、何时才真的改 soul、改的是哪一层。✅ = 已写入画像，💤 = 先记住但还没改五层。

①

首次 init：一次性全量建档

你做了：第一次运行 openbiliclaw init，导入 B站 / 小红书 / 抖音 / YouTube 的历史。

系统怎么走：历史先落成事件 → 跑偏好分析写 preference.json → ProfileBuilder.build 一次性合成五层。

soul 结果：✅ 五层从零全建，直接写 soul.json。这是唯一“无阈值、立刻全建”的流；之后再不会这样大改。

②

看视频 / 账号同步：外层慢慢长

你做了：连刷 5 个“机器学习”长视频（或后台同步了你的 B站 历史）。

系统怎么走：每条先落 SQLite；账号同步直接走 analyze_events，普通浏览进增量管线分到 Surface + Interest 缓冲，攒够阈值（Surface 3 条/5 分钟、Interest 3 条/10 分钟）才更新。

soul 结果：✅ Surface 把 depth_preference 调向长视频；Interest 里“机器学习”权重上升。💤 Role/Values/Core 这次纹丝不动。

③

点开一条推荐：强信号，门槛降到 1

你做了：点开了阿B 推给你的一个冷门纪录片。

系统怎么走：recommendation_click 是强信号，让那一层 min_signals 临时降到 1（仍受 min_interval 约束）；同时进探索缓冲 +0.25。

soul 结果：✅ 比普通浏览更快把这个方向并入 Interest / Surface，下次更敢推同类。💤 仍不碰深层。

④

点 like / dislike：先记住，够了才重建

你做了：给一条推荐点了 dislike。

系统怎么走：① 立刻写 cognition_updates.json（“记住你不喜欢这条”）——此刻 soul.json 没变；② 新反馈攒够 3 条才重析偏好，只有偏好显著变化（高权兴趣变动 / 新增 dislike）才 ProfileBuilder 重建五层。

soul 结果：💤 点完马上“被记住”≠画像已改 → 够阈值后才 ✅ 重建。dislike 还会顺手清掉候选池里的同类内容。

⑤

聊天说一句：重复 + 高置信才沉淀

你做了：聊天里说“我最近在准备考研”。

系统怎么走：DialogueInsightAnalyzer 提取成 goal 候选写 insight_candidates.json。第一次只是候选，不写人格；等再提到（confidence ≥ 0.8 且同类出现 ≥ 2 次）才转成 dialogue_insight 事件进偏好分析。

soul 结果：💤 单独一句不会立刻定人设 → 满足重复+高置信后 ✅ 可能更新 Role 的 current_phase=“备考”。（state 类深层结论门槛更高，需 ≥ 0.9）

⑥

探针点“确认”：确认 ≠ 当场写入

你做了：阿B 猜你可能喜欢“桌游”，弹出兴趣探针，你点了确认。

系统怎么走：API 把该 speculator 标成 confirmed + 后台 force_tick()（只处理探针状态/生成新探针，不写 soul.json）。真正合入要等下一次 pipeline tick 的 _run_speculator_tick → merge_confirmed_interest。

soul 结果：💤 点确认那一刻 soul.json 还没加“桌游” → 等周期 tick 才 ✅ 并进 Interest.likes。避雷探针同理，但只认显式负向（quick_exit 不算）。

⑦

手动改画像：覆盖层，最高优先级

你做了：在画像页手动删掉“游戏”这个兴趣。

系统怎么走：写 profile_overrides.json 立即生效；读取时 get_profile = AI 画像 + 覆盖层。

soul 结果：✅ 你看到的有效画像立刻去掉“游戏”，且 AI 之后重建也不会覆盖你的手动决定；💤 soul.json 里的 AI 原始画像本身不动。若新增的是 dislike，后台异步清池。

3. 关键步骤注释：目标 / 处理 / 写入

这部分是主图的文字版注释，方便核对每一步到底在干什么。

01

事件落库：先保存事实，不急着改画像

目标：把用户行为变成可追溯的事实记录，避免后续 LLM 只能凭印象推断。

处理：MemoryManager.propagate_event() 校验事件类型后写入 SQLite event 层；它本身不会自动向上刷新五层画像。

写入：事件表。后续由 analyze_events() 或 ProfileUpdatePipeline 再消费。

02

偏好分析：把一批事件压缩成结构化偏好

目标：从标题、上下文、反馈和来源里提取稳定的兴趣、风格、上下文、避雷和 UP 主。

处理：PreferenceAnalyzer.analyze_events() 会按 prompt 预算分片；默认过滤 quick_exit 这类被动 negative，保留显式 dislike / thumbs_down 作为避雷证据。

写入：preference.json。这里是高频变化层，不一定立刻重写 soul.json。

03

初始画像 / 重建画像：把材料合成五层 OnionProfile

目标：把长期历史、结构化偏好、近期觉察和洞察假设合成“这个人是谁”的分层画像。

处理：ProfileBuilder.build() 调 LLM 输出严格 JSON；再由 OnionProfile.from_legacy() 和 populate_from_flat_preference() 补齐五层结构。

写入：soul.json、soul_profile.json、soul_profile.md。坏 JSON 或无效画像不会覆盖旧画像。

04

增量管线：先分类、缓冲，再按层刷新

目标：让日常新增信号按影响层级慢慢更新，不让单次点击把深层画像带偏。

处理：ProfileUpdatePipeline.ingest() 把信号映射到 Surface / Interest / Role / Values / Core；每层有 min_signals、min_interval 和 max_buffer。

写入：就绪层会调用 LayerUpdaters；未就绪信号留在 pipeline_state.json。

05

分层更新：外层快变，内层慢变

目标：按证据强度更新不同层：Surface 处理消费风格，Interest 处理主题偏好，Role/Values/Core 处理更长期的身份和心理结构。

处理：Surface 纯计算；Interest 复用偏好分析并把 flat preference 填回兴趣树；Role/Values/Core 走 LLM delta，并带 diff-protection。

写入：层变更写回 soul.json 并追加 soul_changelog.md。注意 portrait 有两条重写入口：① 此增量路径下，仅 Values/Core 层变化触发；② 偏好显著变化时由 ProfileBuilder 整体重建（见步骤 06 与右下结论卡）。

06

推荐反馈：先给 UI 一个“刚记住了什么”

目标：用户点 like / dislike / comment 后，前端能马上看到系统记住了这件事。

处理：record_immediate_feedback_cognition() 只写轻量 cognition card；后台 process_feedback_batch_if_needed() 默认等新反馈累计到 3 条再重析偏好。

写入：即时写 cognition_updates.json；批量路径可能写 preference.json，显著变化时才重建 soul.json。

07

聊天学习：候选先沉淀，重复出现才长期学习

目标：聊天里经常有一次性表达，所以系统先收集候选，不把一句话直接写成人格结论。

处理：DialogueInsightAnalyzer 提取 interest / dislike / goal / value / state；同类候选按 kind + content 合并。进入长期学习需 confidence ≥ 0.8 且出现 ≥ 2 次；即时展示 cognition card 门槛不同：goal/dislike/interest/value ≥ 0.8，唯 state 需 ≥ 0.9。

写入：候选写 insight_candidates.json；合格后转成 dialogue_insight 事件，进入偏好分析和可能的画像重建。

08

觉察 / 洞察周期：给画像重建准备解释材料

目标：不只是知道用户“喜欢什么”，还要记录近期趋势、情绪猜测和可能的深层假设。

处理：CognitionCycle.run_if_due() 默认 12h 运行一次；先生成 awareness，再基于 awareness + preference + soul 生成 insight。

写入：awareness.json、insight.json，并同步到 profile 的 recent_awareness / active_insights 窗口。

09

正向兴趣探针：猜测、验证，再晋升

目标：主动试探用户可能会喜欢但尚未明确表现出来的方向，避免推荐只困在旧兴趣里。

处理：InterestSpeculator 生成 active probes；事件命中或用户确认会提高 confirmation。达到阈值后，主写回路径应由 pipeline tick 合入 Interest.likes。

写入：speculative_state.json；合入画像时调用 merge_confirmed_interest()。当前 API confirm 不是同步写 soul.json，见下方边界说明。

10

避雷探针 / 手动 dislike：确认后同步避雷并清池

目标：只把明确、可确认的不喜欢写入长期过滤，避免把 quick_exit 这类弱负向误当成长期厌恶。

处理：AvoidanceSpeculator 只消费显式负向证据；用户手动新增 dislike 则直接走覆盖层并调度后台清池。

写入：apply_new_dislikes() 会写 preference.disliked_topics，同步 interest.dislikes，并清理候选池。

4. 事件影响矩阵

读法：左边是事件，右边是它会立即动什么、什么时候进入长期画像。

5. 五层画像的变化速度

越往内层越慢，越需要重复、明确、高置信证据。

6. 四个关键结论

看懂这四点，就能判断某个动作会不会马上改 soul。

事件层不等于画像层

MemoryManager.propagate_event() 主要负责落事实。真正画像更新要么走 analyze_events()，要么走 ProfileUpdatePipeline。

即时 cognition 不是重建画像

单条反馈和高置信聊天会先写 cognition_updates.json，用于前端展示“刚记住了什么”，不等于立刻改 soul.json。

画像重建有显著变化门槛

反馈和聊天先更新偏好。仅当高权重兴趣（weight ≥ 0.6）新增/移除 ≥ 2 项、同项权重差 ≥ 0.2，或新增 ≥ 1 个 dislike 时，才由 _preference_changed_significantly 触发 ProfileBuilder 整体重建五层。这与增量路径里 Values/Core 层变化重写 portrait 是两条独立入口。

手动编辑是覆盖层

用户编辑写入 profile_overrides.json，读时叠加在 AI 画像上。画像重建不会覆盖用户手动决定。

当前实现边界：兴趣探针确认不是同步写入 soul.json

源码里的正式合入点是 ProfileUpdatePipeline._run_speculator_tick() 调用 merge_confirmed_interest()。而 /api/interest-probes/respond 的 confirm 分支会先把 speculator 状态标记为 confirmed，并后台调用 force_tick()；这个后台调用本身只处理 speculator state / 新探针生成，不直接把返回的 promoted interest merge 进 soul.json。所以图里把兴趣探针标成“应合入 Interest.likes”，而不是“点击确认后同步已写入”。
