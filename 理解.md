# Openbiliclaw2 理解笔记

## init 数据获取流程

### 数据来源

| 平台 | 获取方式 | 数据类型 |
|---|---|---|
| **B站** | 直接 API 调用 + Cookie 认证 | 观看历史、收藏夹、关注列表 |
| **小红书** | 浏览器扩展同步 | 点赞、收藏、笔记 |
| **抖音** | 浏览器扩展同步 | 观看历史、点赞 |
| **YouTube** | 浏览器扩展同步 | 观看历史、订阅 |

### B站 API 详解

#### 观看历史
- **API**: `GET /x/web-interface/history/cursor?type=archive`
- **认证**: WBI 签名 + 登录 Cookie
- **分页**: 游标分页，循环获取直到达到上限
- **数据**: 视频标题、UP主、时长、观看时间

#### 收藏夹
- **API 1**: `GET /x/v3/fav/folder/created/list-all?up_mid={mid}` — 获取所有收藏夹元数据
- **API 2**: `GET /x/v3/fav/resource/list?media_id={id}&pn=1&ps=20` — 遍历每个收藏夹内的视频
- **数据**: 视频标题、所属收藏夹名称、UP主

#### 关注列表
- **API**: `GET /x/relation/followings?vmid={mid}&pn={page}&ps={page_size}`
- **分页**: 每页 50 条，循环获取直到达到上限
- **数据**: UP主昵称、粉丝数、内容类型

### 初始化四阶段流程

```
Stage 1: 拉取数据
├─ B站: 历史 + 收藏 + 关注（直接 API）
├─ 小红书: 浏览器扩展同步
├─ 抖音: 浏览器扩展同步
└─ YouTube: 浏览器扩展同步

Stage 2: 分析偏好
└─ LLM 调用，分析事件 → 提取兴趣关键词、风格偏好、喜欢的 UP 主

Stage 3: 生成灵魂画像
└─ LLM 调用，构建 MBTI、认知风格、底层需求

Stage 4: 生成首轮推荐池
└─ 多策略发现 + LLM 评估，生成推荐内容候选
```

### 命令行参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| `--bilibili-favorite-limit` | 300 | 收藏初始化信号上限，0 表示跳过 |
| `--bilibili-follow-limit` | 100 | 关注 UP 主上限，0 表示跳过 |

### 认证方式

B站 API 调用依赖 **Cookie 认证**，存放在 `data/bilibili_cookie.json`。

Cookie 通过 `resolve_runtime_cookie()` 函数解析，优先级：
1. 运行时浏览器扩展同步的 Cookie
2. 配置文件 `config.toml` 中声明的 Cookie

### embedding_cache 是什么

每条视频/笔记入库前，会用 LLM 把标题、描述等信息转成一个 **向量（embedding）**，用来计算"这条内容和那条内容有多像"。

这个计算开销大，所以把结果存进 SQLite 缓存（`data/embedding_cache.db`），下次直接读，不用重新算。

**删除后**：已入库的内容全部重新计算 embedding，推荐响应会变慢。
